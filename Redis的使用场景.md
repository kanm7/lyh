#### 什么是缓存穿透？怎么解决

- 缓存穿透：查询一个不存在的数据，mysql查询不到数据也不会直接写入缓存，就会导致每次请求都到达数据库
- 解决方法1：缓存空数据
- 解决方法2：布隆过滤器

#### “布隆过滤器放在哪？是本地内存还是 Redis？如果误判了怎么处理？”

提示：

**布隆过滤器**是一种占用空间极小的概率型数据结构，它由一个很长的二进制向量（BitMap）和一系列随机映射函数（Hash）组成。

**它的特性**：如果布隆过滤器判断某个 ID **不存在**，那它**一定不存在**；如果它判断**存在**，那它有较小的概率其实是**不存在**的（这就是误判）。

在我的项目中，我会选择将布隆过滤器部署在 **Redis** 上（利用 Redis 的 Bitmap）。因为我们的项目是分布式的，放在 Redis 可以保证多个微服务节点共享同一套过滤逻辑

我考虑的是将布隆过滤器作为第一道防线，同时使用缓存空对象，这样即使布隆过滤器发生误判，系统也能做出正确的处理。

布隆过滤器的缺点：通常很难删除某个元素。因为在哈希数组上，某些key可能共享某些位置。

#### 什么是缓存击穿？怎么解决

- 给某一个热点key设置了过期时间，当key过期的时候，恰好这时对这个key有大量的并发请求，这些并发请求可能会瞬间把数据库压垮
- 解决方法1：互斥锁，强一致，性能差
- 解决方法2：设置逻辑过期时间，高可用，性能优，不能保证数据绝对一致

------

#### “逻辑过期解决缓存击穿，具体怎么实现？如果有多个线程同时发现逻辑过期，怎么防止都去重建缓存？”

1. **实现方式**：不再利用 Redis 的过期时间（TTL），而是将过期时间封装在数据对象中，手动判断是否过期。
2. **防止多线程重复重建**：利用 **互斥锁（SETNX）** 机制。
3. **非阻塞特性**：发现过期的线程如果抢不到锁，不会等待，而是直接返回旧数据，从而保证了系统的高可用性。
4. 如果有A、B、C三个线程过来访问key，A先抢到锁，那么A就会开启缓存重建，然后A执行返回旧数据，由新线程完成缓存重建，并且释放锁。

------

#### 什么是缓存雪崩？怎么解决

- 缓存雪崩：在同一时段大量的缓存key同时失效或redis服务宕机，导致大量请求到达数据库，带来巨大压力
- 解决方案：

​	1.给不同的key的ttl添加随机值

​	2.利用redis集群提高服务的可用性

​	3.给缓存业务添加降级限流策略    降级可作为保底策略，适用于缓存穿透、击穿、雪崩

​	4.给业务添加多级缓存

#### （需要去补一下Redission的知识）redis作为缓存，mysql中的数据如何与redis进行同步呢？（双写一致性）

“在我的项目中，对于商户详情这种读多写少的场景，我采用的是Cache Aside Pattern（旁路缓存模式），具体策略是先更新数据库，再删除缓存。

为什么删缓存而不是更新缓存？ 因为更新缓存可能涉及复杂的计算，而且如果写多读少，会造成大量的无效更新。删除缓存是一种懒加载的思想，效率更高。

为什么先写库再删缓存？ 相比于‘先删缓存再写库’，这种方案发生数据不一致的概率更低。

关于强一致性（拓展点）： 当然，这种方案只能保证最终一致性。如果业务要求强一致性（比如必须 100% 同步，不能有一毫秒延迟），我会考虑使用 Redisson 的读写锁。 在读的时候加共享锁，写的时候加排他锁，保证写操作时没人能读。但这样会严重牺牲性能，所以在这个项目中，考虑到并发性能，我选择了‘先写库再删缓存’配合‘超时时间兜底’的方案。”

#### Redis做为缓存，数据的持久化是什么做的？

在redis中提供了两种数据持久化的方式：1.RDB、2.AOF

- RDB：文件存储在磁盘中，它是一个压缩过的二进制文件

​	       **实现原理：在指定的时间间隔内，Redis会通过fork一个子进程，将内存中的数据生成一个快照并写入磁盘**

- AOF：存储在磁盘上，它是一个文本日志文件

​	       **实现原理：每当Redis执行一条写命令，这条命令就会被追加到AOF文件中**



#### 假如Redis的key过期后，会立即删除吗？

Redis对数据设置数据的有效时间，数据过期以后，就需要将数据从内存中删除掉。可以按照不同的规则进行删除，这种删除规则就被称为删除策略（数据过期策略）1.惰性删除、2.定期删除



#### 数据淘汰策略

- Redis提供了8种不同的数据淘汰策略，默认是noeviction不删除任何数据，内存不足直接报错
- LRU：最近最少使用。用当前时间减去最后一次访问时间，这个值越大则淘汰优先级越高
- LFU：最少频率使用。会统计每个key的访问频率，值越小淘汰优先级越高
- 具体的淘汰策略选择，要根据具体使用场景来分析

原生的 `setnx` 锁无法解决“业务没跑完，锁过期了”或者“机器宕机，锁没释放”的问题。

在redission的分布式锁中，提供了一个WatchDog（看门狗），一个线程获取锁成功以后，WatchDog会给持有锁的线程续期（默认是每隔10秒续期一次）

#### Redission的这个锁，可以重入吗？

可以重入，多个锁重入需要判断是否是当前线程，在redis中进行存储的时候使用的hash结构，来存储线程信息和重入的次数

#### Redission锁能解决主从数据一致的问题吗？

“不能。 因为 Redis 的主从复制是异步的。如果 Master 拿到了锁但还没同步给 Slave 就宕机了，Slave 升级为 Master 后，另一个线程也能拿到锁，导致锁失效。

虽然理论上可以用 RedLock（红锁） 解决，它要求在多数节点上加锁成功。但在实际开发中，我们基本不用。 因为红锁性能太差，运维复杂，且算法本身存在争议（Redisson 官方已弃用）。

我的权衡是： 如果业务必须强一致，我会选 Zookeeper。 但在秒杀这种高并发场景下，为了性能，我选择容忍这极低概率的主从切换风险，或者依靠数据库层面的唯一索引来做最终兜底。”

Zookeeper 是强一致的（CP），不会丢锁，但慢；Redis 是高可用的（AP），可能会丢锁，但快。秒杀要快，所以我选 Redis。

#### 面试官：你听说过延时双删吗？为什么不用它呢？

你：“听说过。延时双删主要是为了解决‘先删缓存’策略下的脏数据问题。流程是先删缓存，改库，延时后再删一次。我没有采用它，主要有三个原因：

1.延时时间难把控：需要预估读请求的耗时和数据库主从同步的耗时，很难给出一个精确值。

2.性能损耗：写请求需要休眠或异步等待，增加了系统的复杂度和响应时间。

3.现有方案已足够：我采用的**‘先写数据库，再删缓存’策略，配合超时时间（TTL）**兜底，在绝大多数场景下已经能保证最终一致性。对于非金融级的业务，为了极低概率的并发问题去引入复杂的延时双删，属于过度设计。”

#### 面试官：能解释一下 I/O 多路复用模型吗？

你：

“I/O 多路复用是指利用单个线程同时监听多个 Socket 的技术。

它主要解决了传统阻塞 I/O 中线程无效等待的问题。目前的实现主要基于 Linux 的 epoll 模式。 epoll 的优势在于它采用事件通知机制。当 Socket 就绪时，内核会直接将就绪的 Socket 放入列表通知用户进程，而不需要用户进程去轮询遍历所有 Socket，因此性能极高。

Redis 就是基于 epoll 实现了高效的网络模型。 特别是在 Redis 6.0 引入了多线程 I/O 后，它将命令解析和结果回复这两个耗时的网络操作改成了多线程，而核心的命令执行依然保持单线程。这样既保持了 Redis 的线程安全性，又突破了网络 I/O 的瓶颈。”

#### 面试官：怎么保证 Redis 的高并发高可用？

你：“这个问题要分两方面来看：

第一，保证高可用，我主要使用 哨兵模式（Sentinel）。它能监控集群状态，当 Master 宕机时自动将 Slave 提升为 Master，并通知客户端，实现故障自动转移。

第二，保证高并发，特别是写高并发和海量数据，我会使用 Redis Cluster（分片集群）。它通过 Hash Slot（哈希槽） 将数据分散到多个 Master 节点上，实现了分布式的读写，突破了单机的性能瓶颈。同时，Cluster 自身也具备故障转移能力，不需要额外部署哨兵。所以总结来说：哨兵解决高可用，分片集群解决高并发。”

#### Redis 的主从同步分为两个大阶段：**全量同步**（初始化）和 **增量同步**（正常运行中）。

##### 1. 全量同步：第一次见面的“大工程”

当一个从节点（Slave）第一次连接到主节点（Master），或者因为某种原因断开太久导致数据脱节时，会触发全量同步。

1. **建立连接**：从节点发送 `psync` 命令给主节点。
2. **生成快照**：主节点收到命令后，在后台执行 `bgsave`，生成一份当前的 **RDB 文件**（二进制快照）。
3. **缓存新命令**：在生成 RDB 的过程中，主节点并不会停止服务。它会将这段时间内收到的所有**写命令**缓存到内存中的一个缓冲区——**replication buffer**。
4. **传输 RDB**：主节点将 RDB 文件发送给从节点。
5. **加载数据**：从节点清空旧数据，开始加载 RDB 文件到内存。
6. **同步缓冲区**：RDB 加载完后，主节点把 `replication buffer` 里的写命令发给从节点，从节点执行这些命令，达到最终同步。

------

##### 2. 增量同步：平时的“实时更新”

当全量同步完成后，主从之间会维持一个长连接。

1. **命令传播**：主节点每执行一个写命令，就会异步地将该命令发给从节点。
2. **异步特性**：注意！主节点**不会**等从节点确认收到才返回结果给客户端。这就是我们之前讨论“分布式锁丢失”的根本原因。

------

##### 3. 断线重连后的优化：部分重同步（PSYNC）

如果网络抖动，从节点断开了一会儿又连上了，Redis 2.8 以后不再总是进行昂贵的全量同步，而是尝试“补课”：

- **repl_backlog_buffer**：主节点维护一个固定长度的**环形缓冲区**，记录最近的写命令。
- **offset（偏移量）**：主从都会记录各自处理到了哪个位置。
- **逻辑**：从节点回来后说：“我处理到第 1000 号命令了，你现在到哪了？”主节点一看：“我现在到 1100 号了，这 100 个命令还在我的环形缓冲区里，直接发给你补上就行。”

#### 既然单线程听起来像个“瓶颈”，为什么它还这么快？/ Redis为什么这么快？

1. **没有锁的消耗**： 在多线程环境下，为了保证数据安全，必须引入“锁”（如 `synchronized`）。锁的竞争、释放和等待会消耗大量 CPU 资源。Redis 避开了这一点。
2. **没有上下文切换**： CPU 在不同线程间切换时需要保存和加载状态（上下文切换），这非常耗时。单线程直接跑到底。
3. **内存瓶颈而非 CPU**： Redis 是基于**内存**操作的，并且采用IO多路复用模型，它的瓶颈通常是**内存大小**或**网络带宽**，而不是 CPU 的处理速度。

#### **Redis集群**脑裂，该怎么解决呢？

**候选人**：嗯！这个在项目中很少见，不过脑裂的问题是这样的，我们现在用的是Redis的哨兵模式集群的。

有的时候由于网络等原因可能会出现脑裂的情况，就是说，由于Redis master节点和Redis slave节点和Sentinel处于不同的网络分区，使得Sentinel没有能够心跳感知到master，所以通过选举的方式提升了一个slave为master，这样就存在了两个master，就像大脑分裂了一样，这样会导致客户端还在old master那里写入数据，新节点无法同步数据，当网络恢复后，Sentinel会将old master降为slave，这时再从新master同步数据，这会导致old master中的大量数据丢失。

关于解决的话，我记得在Redis的配置中可以设置：第一可以设置最少的slave节点个数，比如设置至少要有一个从节点才能同步数据，第二个可以设置主从数据复制和同步的延迟时间，达不到要求就拒绝客户端的写请求，就可以避免大量的数据丢失。

------

#### **Redis的**分片集群有什么作用？

**候选人**：分片集群主要解决的是海量数据存储的问题，集群中有多个master，每个master保存不同数据，并且还可以给每个master设置多个slave节点，就可以继续增大集群的高并发能力。同时每个master之间通过ping监测彼此健康状态，就类似于哨兵模式了。当客户端请求可以访问集群任意节点，最终都会被转发到正确节点。

#### **Redis**分片集群中数据是怎么存储和读取的？

**候选人**：

嗯~，在Redis集群中是这样的：

Redis 集群引入了哈希槽的概念，有 16384 个哈希槽，集群中每个主节点绑定了一定范围的哈希槽范围，key通过CRC16校验后对16384取模来决定放置哪个槽，通过槽找到对应的节点进行存储。

取值的逻辑是一样的。

#### 面试官：如果 RabbitMQ 丢失了数据，怎么办？

你：“为了防止消息丢失，我构建了全链路的可靠性保障：

#### 面试官：Lua 脚本是怎么保证原子性的？

你：“这主要得益于 Redis 的单线程模型。

Redis 会将整个 Lua 脚本作为一个不可分割的整体来执行。在脚本执行期间，Redis 不会处理其他任何客户端的请求。这就意味着，脚本内的多条命令（如判断库存、扣减库存）是连续执行的，中间不会被其他线程打断，从而天然地保证了操作的原子性。”

------

#### “你说用 Lua 脚本优化秒杀，能说说 Lua 脚本具体做了什么吗？为什么用 Lua 而不用 Redis 事务？”

如果面试官问起，你可以分三步回答：

1. **具体做了什么**：Lua 脚本将“查询库存、校验一人一单、扣减库存、写入订单”这四个步骤整合为一个原子操作。
2. **为什么用 Lua**：核心在于 Lua 可以在 Redis 内部执行**逻辑判断**，保证了多步操作的**原子性**，解决了高并发下的超卖和并发安全问题。
3. **为什么不用事务**：Redis 事务（Multi/Exec）**不支持逻辑判断**（不能根据上一步的结果决定下一步），且不具备类似关系型数据库的强一致性回滚能力。

------

#### 为什么要用 Lua 脚本来释放锁？如果不使用 Lua 会有什么风险？

**回答要点**：核心在于保证**“判断线程标识”**和**“删除锁”**这两个操作的**原子性**。

**详细分析**：在 `SimpleRedisLock.java` 中，释放锁需要先判断锁是否属于当前线程。如果不使用 Lua，而是分两步执行（先 `get` 再 `delete`），可能会出现这种情况：线程 A 在判断完自己持有锁后，由于 JVM 垃圾回收（GC）等原因导致线程阻塞，此时锁恰好过期并被线程 B 抢占。等线程 A 恢复执行时，它会直接执行 `delete`，从而**误删了线程 B 的锁**。而 Lua 脚本在 Redis 中是原子执行的，能完美避免这种“误操作”。

#### 面试官：为什么不用 RabbitMQ？消息丢了怎么办？

我选择 Redis Stream 是因为它轻量且够用。项目本身就依赖 Redis，直接复用可以降低架构复杂度和运维成本。而且 Stream 支持消费者组和 ACK，性能和功能都满足秒杀需求。

关于消息丢失：

1.消费端不丢：我利用了 Stream 的 Pending List (PEL) 机制。如果消费者处理失败或宕机，消息会停留在 PEL 中。消费者重启后会重试处理这些消息，直到确认成功，保证了至少消费一次。

2.服务端不丢：Redis 支持 RDB/AOF 持久化，重启后数据还在。虽然主从切换可能有极小概率丢数据，但在秒杀场景下，为了高性能，这是我们接受的权衡。”

#### 面试官：你在这个项目中是怎么处理异步下单的？/ 为什么选择这种消息队列？

在项目初期，为了保持架构的轻量级，我直接复用了 Redis。 我利用 Redis 5.0 的 Stream 数据结构实现了异步秒杀。它支持消费者组和 ACK 机制，并且内置了PendingList机制，能够保证消息不丢。同时，redis支持RDB/AOF持久化，重启后数据还在，虽然主从切换可能有极小概率丢数据，但在秒杀场景下，为了高性能，是可以接受的。

 后来，随着我对分布式系统深入学习，我意识到 Redis Stream 虽然好用，但在消息堆积能力和数据可靠性上不如专业的消息队列。 所以，我引入了 RabbitMQ 对异步下单模块进行了重构。具体的改进点如下：

1.更强的可靠性： 我使用了 RabbitMQ 的 ConfirmCallback（发布确认） 和 ReturnCallback（消息回退） 机制，确保消息一定能发送到队列中。 在消费端，我开启了 手动 ACK，只有订单真正写入数据库后才确认消息，防止丢失。

2.死信队列（DLX）兜底： 针对下单失败或超时的异常订单，我配置了死信交换机（Dead Letter Exchange）。处理失败的消息不会直接丢弃，而是进入死信队列，后续可以通过人工介入或定时任务进行补偿，这在 Redis Stream 中实现起来比较麻烦。

3.流量控制（QoS）： RabbitMQ 支持 prefetch count（预取值），可以精确控制每个消费者每次拉取的消息数量（比如每次只拉 1 条），防止瞬间流量把数据库打挂。

#### 如何保证消息不丢失？（RabbitMQ 版）

答：“我做了三层保障：

1.发送端：开启 Publisher Confirm 和 Publisher Returns，确保消息到达交换机和队列。

2.存储端：将 Queue 和 Message 都设置为 Durable（持久化），防止 MQ 重启丢失。

3.消费端：关闭自动 ACK，业务处理成功后手动调用 channel.basicAck()。”

#### 既然换了 RabbitMQ，那 Redis 还在用吗？

答：“还在用！Redis 依然负责最核心的‘前置拦截’。 Lua 脚本依然在 Redis 里跑，负责扣减库存和校验资格。只有校验通过的请求，才会发送到 RabbitMQ。 如果没有 Redis 挡在前面，所有流量直接打到 RabbitMQ，MQ 也会扛不住的。所以是 Redis（抗并发） + RabbitMQ（削峰保底） 的组合。”

#### 看门狗原理是什么？

“关于看门狗： 它是 Redisson 的自动续期机制。 当我不指定锁的过期时间时，Redisson 默认给锁 30 秒过期，并启动一个后台线程，每隔 10 秒检查一次。如果业务还在运行，就重置过期时间。这既防止了业务没跑完锁就失效，又防止了宕机死锁。

#### 面试官：BitMap 和 GEO 的底层数据结构是什么？

BitMap 的底层是 String。 它利用了 String 类型底层的字节数组，通过位操作命令（如 SETBIT）来单独操作每一个 bit 位，非常节省空间。

GEO 的底层是 ZSet（有序集合）。 它利用 GeoHash 算法将经纬度编码成一个整数作为 ZSet 的 Score，将商铺 ID 作为 Member。这样就可以利用 ZSet 的排序和范围查询功能来实现‘附近的人’或‘附近商铺’的搜索。

#### 如何保证消息的顺序性？

RabbitMQ 只能保证单个队列+单个消费者的场景下消息是顺序的。一旦开启了多消费者并发，就无法保证顺序。如果业务必须保证严格的顺序（比如同一个用户的订单状态变更），我的解决方案是：

- 分发到不同队列：在生产端，根据业务 ID（比如 userId）进行哈希，将同一个用户的消息固定发送到同一个队列中。
- 单消费者处理：保证每个队列只被一个消费者监听。这样，虽然牺牲了部分并发性，但保证了特定业务场景下的严格顺序

#### 如何处理消费端的重复消费问题？

面试官心理：考察你对幂等性的理解。

回答：“因为网络重试等原因，重复消费是可能发生的。为了解决这个问题，核心是保证消费逻辑的幂等性。我主要用了两种方法：

1.数据库唯一索引：在订单表上给订单号创建一个唯一索引。即使有重复的消息进来，第二次 INSERT 时数据库会直接报错，保证了数据不会重复。

2.Redis 分布式锁：在消费消息时，先用消息的唯一 ID（比如订单号）去 Redis SETNX。如果能设置成功，就执行业务逻辑；如果失败，说明正在被处理或已经处理过，直接丢弃即可。”

#### RabbitMQ 有哪几种交换机类型？

面试官心理：考察你对路由机制的理解。

回答：“主要有四种：

1.Direct Exchange（直连）：根据 RoutingKey 完全匹配来投递。我项目中的订单处理就是用的这种，一个 Key 对应一个队列，非常明确。

2.Fanout Exchange（扇出）：忽略 RoutingKey，广播给所有绑定的队列。适合发通知、公告的场景。

3.Topic Exchange（主题）：支持通配符（* 和 #）进行模糊匹配。非常灵活，适合复杂的订阅场景。

4.Headers Exchange（头）：根据消息头中的属性来匹配，不常用。”

#### ThreadLocal•项目场景：用户登录拦截器 (LoginInterceptor)。

回答：使用场景：在我的项目中，我使用 ThreadLocal 来存储当前登录用户的信息（UserDTO）。 当请求进入拦截器时，我解析 Token 获取用户信息，存入 ThreadLocal。这样在后续的 Service 层（如下单、点赞）中，我可以直接通过 UserHolder.getUser() 获取当前用户，避免了参数层层传递。

底层原理：ThreadLocal 内部维护了一个 ThreadLocalMap。它的 Key 是 ThreadLocal 实例本身（弱引用），Value 是我们存的对象（强引用）。每个线程都有自己独立的 Map，所以实现了线程隔离。内存泄漏：因为 Key 是弱引用，GC 会回收 Key，但 Value 是强引用不会被回收，导致 Key 为 null 但 Value 还在，造成内存泄漏。 

解决方法：我在拦截器的 afterCompletion 方法中，显式调用了 UserHolder.remove()（即 ThreadLocal.remove()），在请求结束时清理数据，防止内存泄漏。

#### Feed 流的推送模式及大 V 问题

**推模式**也称为“写扩散”。其核心思想是：**发布者发布内容时，主动将内容推送到所有粉丝的收件箱中**。

**拉模式**也称为“读扩散”。其核心思想是：**发布者只管发，粉丝刷新时才去“拉取”关注列表里所有人的新动态**。

**回答要点**：你使用了 **推模式 (Push)**，将笔记 ID 写入粉丝的收件箱。

**详细分析**：大 V 粉丝量巨大（如千万级），推模式会造成极大的写压力和存储浪费。在工业界，通常对大 V 采用“拉模式”或“推拉结合”，即普通人发推，大 V 只写自己的发件箱，粉丝上线后再去拉取。

小博主用推 大博主用拉

#### 大 Key 和热 Key 怎么保证业务不崩？

**大 Key (Value 很大)：** 拆分！例如，把一个大的 List 拆成 `list:1`, `list:2`。或者定期清理。

**热 Key (访问量极高)：** 

1. **本地缓存：** 在应用层（Java 堆）存一份，不查 Redis。 
2. **副本分散：** 给热 Key 加随机后缀，存到不同的 Redis 节点上。
