# 如何应对突发大流量

面对突发大流量（比如秒杀、大促）的场景，我们的系统就像是一座面临洪峰的大坝。为了不让大坝溃决，我们需要一套**多层次的防御体系** 🛡️。

在大厂面试中，处理大流量的核心思路可以概括为四个字：**“限、削、缓、稳”**。

1. **限（限流/丢弃）**：利用 **Sentinel** 或网关在入口处把超过承载能力的流量直接挡掉，保证系统不宕机。
2. **削（削峰填谷）**：利用 **MQ** 将瞬间涌入的请求变成平滑的流，让后端服务按自己的节奏处理。
3. **缓（缓存抗压）**：利用 **Redis** 把最沉重的数据库查询压力扛下来。
4. **稳（熔断降级）**：如果某个次要环节挂了，直接断开，确保主交易链路依然稳定。

**Redis 预扣库存与 Lua 脚本** ⚡ 这是抗住“秒杀”第一波冲击的核心逻辑。探讨如何不在数据库扣库存，而是利用 Redis 的高性能和 Lua 脚本的原子性来瞬间处理高频请求。

**MQ 削峰填谷的实战细节** 🌊 深入研究当几十万个请求涌入时，消息队列是如何充当“蓄水池”的，以及如何处理可能产生的消息积压或丢失问题。

**多级缓存策略（从 CDN 到本地缓存）** 🧊 了解如何让流量在到达你的服务器之前就被层层“过滤”掉。比如什么是 CDN 静态加速，以及为什么要引入 Caffeine 等本地缓存。

## 一、Redis 预扣库存与 Lua 脚本

### 1. 为什么数据库（DB）会成为瓶颈？ 🐢

在平时的业务中，扣减库存通常是：

1. `SELECT` 查询库存是否足够。
2. `UPDATE` 修改库存数量。

但在秒杀这种突发大流量下，成千上万的请求同时涌入，MySQL 的**行锁机制**会导致请求排队。数据库的 QPS（每秒处理请求数）通常在几千级别，一旦超过这个限度，数据库连接池会被占满，整个系统就会像多米诺骨牌一样倒下，这就是我们常说的“雪崩”。

### 2. Redis 预扣库存：把压力扛在前面 🛡️

为了保护数据库，我们把库存数据提前“同步”一份到 **Redis** 中。

- **Redis 的优势**：它是基于内存的操作，单机 QPS 可以达到 10 万+。
- **逻辑转变**：当请求进来时，不再直接去操作 MySQL，而是先在 Redis 里扣减。
  - 如果 Redis 扣减成功，说明“抢到了”，再异步写数据库。
  - 如果 Redis 提示库存不足，直接给用户返回“已售罄”，请求根本不会到达数据库。

### 3. Lua 脚本：解决“超卖”的原子锁 🔒

这里有一个非常棘手的问题：**并发安全**。 如果我们在 Java 代码里先 `get` 库存，再 `set` 扣减后的值，在高并发下依然会出现“超卖”：两个请求同时查到库存剩 1，结果都执行了减 1，库存变成了 -1。

为了解决这个问题，我们会使用 **Lua 脚本**。

- **原子性**：Redis 执行 Lua 脚本是原子的。这意味着整个“查询+扣减”的逻辑在 Redis 执行期间不会被其他请求打断。
- **逻辑封装**：我们将“判断库存是否足够”和“执行减法”写在一个 Lua 脚本里，交给 Redis 一次性完成。

### ✍️ 思考一下

我们已经用 Redis 扛住了第一波流量冲击，并且保证了库存不会扣成负数。

但请设想这样一个场景：如果你在 Redis 里成功扣减了库存，但在后续异步通知数据库写订单的过程中，你的服务器突然宕机了。这时候，Redis 里的库存减了，但数据库里并没有订单（**少卖问题**）。

**这种“数据不一致”的情况，你觉得会对业务产生什么影响？我们应该如何优化来保证库存最终是准确的呢？** 🤔

------

在传统的数据库方案中，超卖通常是因为多个线程**同时修改**同一行数据导致的。而我们选择的 **Redis 预扣减** 方案，正是为了在“保住数据库”的同时，解决超卖问题。

### 🛡️ Redis 是如何防止超卖的？

我们之所以敢把库存放在 Redis 里先扣，是因为 Redis 有两个杀手锏：

1. **单线程原子性** 🧵：Redis 核心处理逻辑是单线程的。这意味着对于同一个 key 的操作，必须排队一个一个来，天然避免了并发冲突。
2. **Lua 脚本** 📜：我们将“查询剩余库存”和“扣减库存”这两步操作写在一个 Lua 脚本中。Redis 会把整个脚本看作一个不可分割的整体（原子操作）。要么全部成功，要么全部失败，绝不会出现“两个请求同时发现库存剩 1，结果都扣减成功”的情况。

### ⚠️ “少卖”与“最终一致性”

虽然我们解决了超卖，但正如你之前担心的，如果 Redis 扣成功了，但后端创建订单失败了，就会出现**“少卖”**（Redis 剩 0，实际 DB 还有 2）。

为了解决这个“副作用”，我们需要一套**补偿机制**：

- **回滚库存** 🔄：如果后续的业务逻辑（写数据库）执行失败，程序需要捕捉异常，并向 Redis 发送一条指令，把刚才扣掉的库存“还回去”。
- **定时核对** 🕵️：系统会定期对比 Redis 和数据库里的库存数据。如果发现 Redis 里的数长期小于数据库，说明有“虚扣”发生，会自动进行修正。

### 💡 老师的引导提问

既然我们已经用 **Redis + Lua** 挡住了洪峰并解决了超卖，那么剩下的那几十个“幸运儿”最终还是要进入数据库去创建订单。

如果这几十个请求**同时**去写数据库的 `order` 表，为了提高效率，我们通常不会让交易服务同步等待数据库写入完成。

**你觉得我们应该用什么技术，能让这几十个请求“排好队”慢慢写入数据库，而不至于让数据库瞬间压力过大？** 🤫（提示：我们之前聊过一个可以“削峰填谷”的中间件。）

## 二、MQ：系统的“蓄水池” 💧

我们可以把 **MQ (消息队列)** 想象成一个大型的蓄水池。

- **同步变异步**：交易服务（Trade Service）在 Redis 预扣成功后，不再同步等待数据库写订单。它只需要把“某某用户买了某某商品”的消息随手丢进 MQ，就立刻给用户返回“抢购中...”。
- **保护后端**：数据库服务（Order Service）像是一个口径比较小的水管，它会根据自己处理的能力（比如每秒处理 100 个），稳扎稳打地从 MQ 里抓取消息进行落盘。

### 这里的关键问题 🧠

虽然这种方式极大地保护了数据库，但也带来了一个业务上的尴尬：**“异步化”后的用户反馈**。

在传统的同步流程中，用户点完“下单”，要么成功要么失败，一秒钟见分晓。但在 MQ 异步流程下，用户点完按钮，后台可能还在排队写数据库呢。

------

**既然数据库落盘是异步的，用户下单后，我们该如何告诉他到底“抢购成功”了没有呢？总不能让用户一直看着“抢购中”发呆吧？** 🤔 你觉得在实际的电商 APP（比如淘宝、京东）里，它是怎么处理这个反馈的？

很多时候为了追求极致的速度，系统在 **Redis** 扣减成功后会立刻给用户返回“下单成功” 🎊。但这里其实隐藏着一个有趣的挑战：

在异步流程中，**Redis 扣减成功** 并不百分之百等于 **数据库下单成功**。如果消息在 **MQ** 里丢了，或者数据库因为某种极端情况宕机了，用户以为抢到了，但后台其实没单子，这就造成了“订单丢失”的严重事故 😱。

为了平衡“快”和“准”，大厂通常有以下几种处理用户反馈的方案：

### 异步下单的反馈策略 🛠️

| **方案**                    | **做法**                                                     | **优缺点**                                   |
| --------------------------- | ------------------------------------------------------------ | -------------------------------------------- |
| **轮询 (Polling)** 🔄        | 前端收到“请求已受理”后，每隔 1 秒调用一个“查询订单状态”的接口。 | 实现简单，但会对服务器造成额外的查询压力。   |
| **长连接 (WebSocket)** 📡    | 建立一个双向通道，后台数据库写完后，主动推送消息给前端。     | 实时性最高，但维护长连接比较消耗服务器资源。 |
| **乐观反馈 (Optimistic)** 🚀 | 就像你说的，直接返回成功。如果后续真的失败了，再通过短信或站内信通知用户。 | 用户体验最爽，但处理后续失败的逻辑非常复杂。 |

在实际的秒杀页面，你可能经常看到一个进度条在转，提示“正在排队中...” ⏳。这其实就是前端在背后不停地**轮询**后端，询问：“我的单子写进数据库了吗？”

**让我们深入思考一下：** 如果我们采用“轮询”的方案，成千上万的用户同时在不停地刷接口问“成功了吗？”，这个“查询请求”会不会再次把数据库冲垮？🤔 我们该如何利用 **Redis** 来保护这个查询接口呢？

当成千上万的用户在疯狂点击“查询订单状态”时，如果每一个请求都直接去数据库执行 `SELECT` 查询，数据库依然会因为负载过高而瘫痪。🏥

为了保护数据库，我们通常会把**订单的结果**也提前放在 **Redis** 里。

### 保护查询接口的策略 🛡️

1. **结果预存**：当我们在 Redis 里通过 Lua 脚本成功预扣库存后，我们可以顺便在 Redis 里存入一个“占位符”（比如：`order:status:用户ID = 处理中`）。
2. **异步更新**：等后端的数据库真正写完订单后，我们再去把 Redis 里的状态改为“下单成功”或“下单失败”。
3. **查询闭环**：前端在轮询时，请求只到达 Redis。
   - 如果 Redis 说“处理中”，前端继续转圈圈。⏳
   - 如果 Redis 说“成功”，前端跳转到支付页面。✅
   - 如果 Redis 里没这个状态（可能是超时了），再考虑去查数据库。

这样一来，99% 的查询压力都被 Redis 给扛住了，数据库只需安静地处理那几百个真正的订单写入即可。
