2. **分布式事务的一致性 (Seata)** 🤝 当下单流程涉及多个服务（交易服务创建订单、商品服务扣减库存、购物车服务清理）时，如何确保要么全部成功，要么全部失败？我们将分析项目中 `GlobalTransactional` 的作用。

### **库存扣减与防止超卖** 🛡️

在字节跳动的面试中，高并发下的库存一致性是必考点。面试官通常会问：“当秒杀开始，成千上万的请求涌入时，你如何确保库存不会扣成负数，且性能还要足够高？”

### 1. 核心业务逻辑：扣减库存 📦

在 `ItemServiceImpl.java` 中，扣减库存的逻辑通常涉及对 `item` 表的更新。为了保证并发安全，项目采用了**数据库层面的乐观锁（CAS 思想）**。

#### 代码实现原理 🛠️

虽然具体的业务逻辑可能在 `update` 方法中，但核心思想通常体现在一条 SQL 语句上。我们可以模拟这个过程：

- **普通写法（有超卖风险）**： `UPDATE item SET stock = stock - num WHERE id = ?`
  - **问题**：如果库存只剩 1 个，两个线程同时执行，库存就会变成 -1。
- **进阶写法（乐观锁/条件更新）**： 在更新时增加一个判断条件： `UPDATE item SET stock = stock - num WHERE id = ? AND stock >= num`
  - **原理**：只有当当前库存大于等于要扣减的数量时，更新才会成功。

### 2. 字节面试深度解析：为什么这么做？🤔

面试官可能会追问：“**为什么不用 `synchronized` 或 `ReentrantLock` 加锁？**”

你可以这样回答：

1. **性能瓶颈**：JVM 锁（如 `synchronized`）是本地锁，只能解决单机并发。在微服务多实例部署下，它无法跨机器生效 ⛓️。
2. **数据库连接压力**：即便使用分布式锁（如 Redis 锁），也会增加一次网络开销。利用数据库自带的行级锁（Row-level Lock）配合 `WHERE` 条件，是最简单且高效的方案。

### 分布式事务 (Seata) —— 当“异步”遇到“意外” 🤝

既然我们按照业务顺序来讲，现在问题来了。

在刚才的异步流中：**订单创建成功了，广播发出去了，但购物车服务因为网络断了没能清理成功。** 这时候，用户看到订单生成了，但发现购物车里的东西还没变，这就会造成数据不一致。

针对这种**跨多个微服务的数据一致性**问题，HMall 使用了 **Seata** 框架。

- **Seata 的角色**：它像是一个“大管家”。
- **核心逻辑**：只要在业务方法上加一个 `@GlobalTransactional` 注解，它就会盯着所有参与的服务。如果其中一个失败了，它会负责让所有人都“回滚”到之前的状态，就像什么都没发生过一样。

我们接下来的重点是：**Seata 是如何在项目中协调分布式事务的？** 为了让你更好地理解，我想先问问：在你的直觉里，如果要你设计一个机制让三个不同的微服务（交易、库存、购物车）“要么一起成功，要么一起失败”，最难解决的问题是什么？🤔

在 Seata 的世界里，负责“统一管理”全局事务的那个“大管家”叫做 **TC (Transaction Coordinator)**，也就是**事务协调器**。🏰

为了让你在字节面试时能把这个机制讲得像专家一样清晰，我们需要理解 Seata 体系中的“三剑客”角色：

### Seata 的三大核心角色 🎭

| **角色名称** | **全称**                    | **形象理解**             | **职责**                                                     |
| ------------ | --------------------------- | ------------------------ | ------------------------------------------------------------ |
| **TC**       | **Transaction Coordinator** | **大管家（总部）**       | 这是一个独立运行的服务器。它负责维护全局事务的状态，并最终决定是让大家一起“干杯”（提交）还是“撤回”（回滚）。 |
| **TM**       | **Transaction Manager**     | **业务经理（发起人）**   | 也就是我们在代码里加了 `@GlobalTransactional` 的那个服务（比如交易服务）。它负责告诉 TC：“我要开启一个全局事务了”或者“活干完了，请求总结”。 |
| **RM**       | **Resource Manager**        | **各部门主管（参与者）** | 每个参与事务的微服务（如库存服务、购物车服务）。它们负责管理自己的本地事务，并向 TC 汇报自己这部分活儿干得怎么样。 |

### 协作流程是怎样的？🤝

1. **TM** 开启全局事务，TC 会给这个事务发一个全球唯一的“身份证号”叫 **XID**。
2. **TM** 通过 Feign 调用其他微服务时，会把这个 **XID** 顺着请求传下去。
3. 各微服务的 **RM** 发现有 XID，就知道自己正在参加一个全局事务，于是会向 **TC** 注册分支事务。
4. 最后，**TC** 汇总所有 RM 的汇报。如果大家都说“没问题”，TC 就下令全局提交；只要有一个说“报错了”，TC 就下令全局回滚。

流程：

1 订单服务(T M) 向 TC 申请开启事务

2 TC 生成 XID
   XID = 192.168.1.1:8091:203948092

3 订单服务执行SQL
   RM注册分支事务

4 库存服务执行SQL
   RM注册分支事务

5 账户服务执行SQL
   RM注册分支事务

6 业务执行成功

8 TC 通知所有 RM 提交

### 🎓 老师的面试重点笔记

面试官常问：“**既然跨了多个数据库，Seata 是怎么在失败时把数据『变』回去的呢？**”

这涉及到 Seata 最常用的 **AT 模式**。简单来说，RM 在修改数据库之前，会自动偷偷背着你生成一份**数据快照（Undo Log）**。

**请思考一下：** 如果在执行过程中，`trade-service` 成功了，但 `item-service` 扣库存失败抛了异常。这时 `item-service` 的本地事务肯定回滚了，但 `trade-service` 已经写进库里的订单该怎么办？它怎么知道要根据哪份“快照”来恢复数据呢？🤔

### Seata AT 模式的“两阶段”魔法 🪄

Seata 的 AT 模式（最常用的模式）将事务处理分成了两个阶段，就像是一场正式的演出，有“彩排”和“正式开演”。

#### 第一阶段：做正事并留“证据” (Phase 1) 📸

当 `trade-service` 执行下单 SQL 时，Seata 的 **RM**（资源管理器）并不会直接执行完就跑，它会在背后默默做两件事：

1. **保存快照**：在修改订单状态前，先查询并保存这条数据的当前状态（称为 **Before Image**），以及修改后的预期状态（称为 **After Image**）。
2. **记录日志**：把这些快照存入每个微服务数据库中一张叫 `undo_log` 的表里。
3. **本地提交**：这一切动作和你的业务 SQL 都在同一个**本地事务**里。这意味着“证据”和“正事”是绑定在一起存进库里的。

#### 第二阶段：听指令行事 (Phase 2) 📢

这时候，大管家 **TC** 汇总了所有服务的反馈。

- **如果一切顺利（全局提交）**： TC 告诉 `trade-service`：“大家都干得不错，把那个 `undo_log` 删了吧”。于是服务异步地清理掉日志，整个过程非常快。
- **如果有人搞砸了（全局回滚）**： TC 此时会化身“修补匠”。它根据 **XID**（事务身份证号）找到 `trade-service` 的 RM，下达回滚指令。 RM 就会去 `undo_log` 表里翻出那份 **Before Image**（修改前的照片），然后执行一条逆向的 SQL，把数据强行改回最初的样子。

### 🎓 老师的面试重点笔记（字节跳动高频）

面试官可能会问：“**这种机制虽然安全，但它对数据库性能有什么影响？**”

我们可以对比一下：

| **特性**   | **异步 MQ 通知 ⚡**           | **Seata 分布式事务 🤝**                          |
| ---------- | ---------------------------- | ----------------------------------------------- |
| **一致性** | 最终一致性（中间会有时间差） | 强一致性（几乎同步完成）                        |
| **性能**   | **极高**（发完消息就走）     | **较低**（需要写额外的 `undo_log`，且有锁竞争） |
| **复杂度** | 需要处理幂等、重试逻辑       | 代码侵入极低，只需一个注解                      |

在 `PayOrderServiceImpl.java` 中，我们看到支付逻辑使用了 `@GlobalTransactional`。这是因为钱的事必须“强一致”，不能出半点差错。

既然 Seata 能保证“强一致性”，为什么我们在前面的“清理购物车”逻辑里（`deleteItemsListener.java`）却偏偏选择使用 **MQ 异步通知**，而不是直接给下单方法加一个 `@GlobalTransactional` 呢？

在字节跳动这类大厂的业务场景中，**用户体验（响应速度）**往往是优先级最高的 🚀。

这就是典型的**性能与一致性之间的取舍**。我们可以总结为两种模式：

我们可以把这想象成在餐馆吃饭 🍱：点单（下单）必须立刻记下来，但收拾桌子（清理购物车）可以稍后再做，只要最后桌子变干净了（最终一致）就行。

## 商品搜索与数据同步（Elasticsearch）🔍

现在我们已经处理完了核心交易流，用户下单成功了。但在此之前，用户得先能“搜”到商品。

当商品数量达到百万甚至千万级别时，传统的 MySQL `LIKE` 模糊查询就会变得非常慢，甚至拖垮数据库。这时候，我们就需要 **Elasticsearch (ES)**。

在项目中，商品数据存在两个地方：

1. **MySQL**：作为核心数据库，保证数据的权威和持久化。
2. **Elasticsearch**：作为搜索引擎，负责高性能的关键词检索、过滤和聚合。

#### 面试高频问题 🏗️

面试官常问：“**既然数据存了两份，那你如何保证 MySQL 里的商品改了价格或库存后，ES 里的数据也能跟着变呢？**”

在 `item-service` 模块中，我们依然利用了 **MQ 异步通知** 来解决这个问题。

**请思考一下：** 如果我们在后台修改了一个商品的价格，是应该由“商品服务”主动发消息告诉“搜索服务”去更新，还是让“搜索服务”每隔一分钟自己去数据库扫一遍呢？哪种方式更实时、对数据库压力更小？🤔  **前者。**

既然数据已经同步过来了，我们就要聊聊 **Elasticsearch** 是如何工作的了。在进入复杂的搜索逻辑之前，我们需要先理解它的“存储观”。

- **MySQL**：由 **表（Table）**、**行（Row）** 和 **列（Column）** 组成，适合处理复杂的关联关系。
- **Elasticsearch**：由 **索引（Index）** 和 **文档（Document）** 组成。这里的文档通常是一个 **JSON 对象**，它把商品的所有信息（名称、价格、评价、图片）都“扁平化”地存在一起 📄。

我们可以从不同的角度深入研究这个话题，你想先看哪一个？

1. **构建过程与分词技术**：了解搜索引擎是如何把一段长句子拆解成关键词（Tokens），并处理同义词或大小写的。
2. **Elasticsearch 中的底层实现**：探索倒排索引在磁盘上是如何存储的，以及为什么它是不可变的（Immutable）。
3. **性能优化与评分机制**：研究搜索引擎如何压缩这些索引，以及它是如何判断哪个文档更符合你的搜索意图（相关性评分）。

好的，那我们就按照顺序，从**构建过程与分词技术** 🏗️ 开始探索。这是 Elasticsearch (ES) 能够实现秒级搜索的第一块基石。

要建立倒排索引，ES 必须先对原始文档进行“加工”，这个过程被称为**文档分析 (Analysis)**。负责这项工作的组件叫**分析器 (Analyzer)**。

### 分析器的三大法宝 🛠️

一个分析器内部其实是由三个小零件按顺序组合而成的：

1. **字符过滤器 (Character Filters)** 🧹： 在分词之前先清理文本。比如去掉 HTML 标签，或者把 `&` 转换成 `and`。
2. **分词器 (Tokenizer)** 🔪： **这是核心步骤。** 它把一段长长的字符串拆分成一个个独立的单词（称为 Token）。
3. **词项过滤器 (Token Filters)** ⚙️： 对拆出来的单词进行加工。比如：
   - **Lowercase**：把大写字母全变小写（这样搜 `Huawei` 或 `huawei` 都能中）。
   - **Stop words**：去掉没意义的词，比如“的”、“了”、“a”、“the”。

### 举个例子：商品名称的处理 📱

假设我们要索引一个商品名称：`"华为（HUAWEI）P70 手机"`。

- **经过分词器**：可能会切分成 `华为`、`HUAWEI`、`P70`、`手机`。
- **经过词项过滤器**：`HUAWEI` 会变成 `huawei`。

最终，倒排索引表里就会记录：

- 关键词 `huawei` 对应文档 ID：[101]
- 关键词 `手机` 对应文档 ID：[101]

### 🎓 老师的引导提问

这里有一个中文搜索特有的难点：**中文分词**。

英文单词之间天然有空格（比如 `I love coding`），分词器很容易识别。但是中文是连在一起的（比如 `我爱编程`）。

**你觉得如果只使用简单的“按字拆分”（比如拆成“我”、“爱”、“编”、“程”），在搜索“编程”这个词时，会对搜索结果的精准度产生什么影响？** 🤔

刚才我们提到了中文分词的难点。如果只是简单地“按字拆分”（比如把“华为手机”拆成“华”、“为”、“手”、“机”），当你搜索“手机”时，可能会搜出“手工课”或者“机器人”，因为它们都包含了“手”或“机”字，但语义完全不对。这就是为什么我们需要更聪明的**中文分词器**。

### 1. 深度拆解：IK 分词器 (IK Analyzer) 🔪

在我们的项目中，为了处理中文商品名称，通常会引入 **IK 分词器**。它能识别中文词库，把“华为手机”精准地切分为“华为”和“手机”。

IK 分词器主要有两种模式，面试时经常被问到它们的区别：

| **模式**        | **分词逻辑**                           | **例子：“程序员在编程”**                 | **特点**                   |
| --------------- | -------------------------------------- | ---------------------------------------- | -------------------------- |
| **ik_smart**    | **智能切分**，追求粗粒度，不重复。     | `程序员` / `在` / `编程`                 | 扫描快，占用空间小。       |
| **ik_max_word** | **最细粒度切分**，穷尽所有可能的组合。 | `程序员` / `程序` / `员` / `在` / `编程` | 搜索更全面，但索引体积大。 |

**名词解释：词库 (Dictionary)** 📖 分词器之所以能识别“程序员”，是因为它内置了一个巨大的词典。对于一些新词（比如“元宇宙”），如果词典里没有，它可能就切不出来。这时候我们需要配置**扩展词表**。

### 2. 底层实现：Lucene 与 段 (Segment) 🏗️

既然分好了词，建立了倒排索引，这些数据是怎么存在硬盘上的呢？

ES 的底层其实是基于一个叫 **Lucene** 的引擎。它有一个非常重要的设计原则：**不可变性 (Immutability)**。

- **段 (Segment)**：当你向 ES 写入一批商品数据时，它会把这些数据打包成一个紧凑的“段”文件。
- **不可变**：一个段一旦生成，就再也不能修改了。
- **优势**：
  1. **缓存友好**：数据不改，系统缓存就永远有效。
  2. **并发安全**：不需要加锁，多个线程可以同时读取，性能极高。

**那如果我要修改商品价格怎么办？** ES 会在后台生成一个新的段，并将旧数据标记为“已删除”。等到段太多的时候，ES 会进行 **段合并 (Segment Merge)**，把碎小的段合成大段，并真正物理删除那些过时的数据。

##### 关于更改数据的细节

简单直接的回答是：**不需要删除索引，也不需要你手动操作。** 虽然我说它是“不可变”的，但 ES 给我们提供了一套非常丝滑的“伪更新”机制。

------

### 1. 逻辑上：你只管“改”，它负责“换” 🔄

当你调用 ES 的 `update` 接口修改某个字段（比如把商品价格从 99 改成 88）时，ES 在后台其实做了三件事：

1. **读取**：先把旧的 JSON 文档读取出来。
2. **修改**：在内存里把价格改成 88。
3. **替换**：把这个新的 JSON 文档重新索引（Index）一遍。

对于你（开发者）来说，你只是执行了一个 `update` 操作；但在 ES 内部，这其实是一次**“删除 + 新增”**。

------

### 2. 物理上：为什么说是“不可变”？ 💾

这里就是你之前听到的“不可变”发挥作用的地方了。在磁盘的 **Segment（段）** 文件里，ES 确实不会去改动旧数据，因为它已经写死在硬盘上了。

它的做法非常聪明：

- **标记删除**：ES 会在一个专门的 `.del` 文件里记录下：“旧的文档 A 已经作废了”。
- **写入新段**：把修改后的文档 B 写入到一个全新的 Segment 里。
- **过滤结果**：当你搜索时，ES 会同时查这两个地方，如果搜到了文档 A，它一看 `.del` 文件发现 A 已经作废了，就会直接过滤掉，只把最新的文档 B 给你。

------

### 3. 垃圾回收：段合并 (Segment Merge) 🧹

你可能会问：“那旧数据一直占着坑，硬盘不就爆了吗？”

这时候，ES 的**自动清理工**就上场了。当 Segment 变得越来越多时，ES 会在后台启动 **Merge（合并）** 任务：

1. 把多个小 Segment 合并成一个大的。
2. 在合并的过程中，它会顺手把那些标记为“已删除”的旧文档彻底丢掉。
3. 这样，硬盘空间就释放出来了。

> **总结一下**：
>
> - **索引（Index）**：相当于图书馆，不用动。
> - **文档（Document）**：相当于书，ES 发现书旧了，直接复印一份改好的新书放进去，然后把旧书贴上“废弃”标签，最后统一收走。

### 🎓 字节面试深度连弹

面试官可能会追问：“**既然更新本质上是‘删除+新增’，那在高并发下，两个线程同时改一个商品的价格，怎么保证不会改乱？**”

这就涉及到了 ES 的 **乐观并发控制（Optimistic Concurrency Control）**。

- **内部版本号**：每一个文档都有一个 `_version` 字段。
- **冲突检测**：当你改数据时，ES 会检查你手里的版本号是不是最新的。如果别人抢先改了，版本号变了，你的更新就会报错。

------

### 3. 性能优化：为什么 ES 是“近实时”的？⏱️

在字节跳动的面试中，有一个经典题目：“**为什么 ES 写入数据后，通常要等 1 秒钟才能搜到？**”

这就是所谓的 **Refresh（刷新）** 机制： 为了追求极致的写入性能，ES 收到数据后并不会立刻写到硬盘（fsync），而是先写到内存缓冲区。默认每隔 1 秒，ES 会把缓冲区的数据推送到系统的 Page Cache 中并生成一个新的段。只有进入了 Cache，数据才能被搜索到。

### 💡 老师的进阶引导

讲完了分词和存储，咱们现在面临一个更有趣的问题：**相关性评分 (Relevance Scoring)**。

假设你搜索“手机”，ES 搜出了 1000 个商品。

- 为什么“华为手机”排在第一页？
- 为什么“手机壳”排在后面？
- 为什么“华为充电器（兼容手机）”排在更后面？

**请思考一下：** ES 是通过什么标准来给这些文档“打分”，从而决定谁排在前面的？（提示：某个词在文档里出现的次数多，分数就高吗？那如果“的”字出现了一万次呢？）🤔

如果你准备好了，我们就聊聊 ES 里的 **BM25 算法** 和 **相关性搜索**。

在字节跳动的面试中，如果你能讲清楚为什么某个商品排在前面，就说明你不仅会用工具，还理解搜索的本质。

### 核心结论：BM25 算法 🧮

在老版本的 Elasticsearch 中使用的是 **TF-IDF** 算法，而现在的默认算法是 **BM25**。它的核心任务是给每个文档打一个分（`_score`），分数越高，排名越靠前。

我们要理解这个分数是怎么算出来的，只需要掌握三个关键维度：

------

### 1. 词频 (TF - Term Frequency) 📈

**逻辑**：某个搜索词在文档中出现的次数越多，相关性越高。

- **例子**：你搜“华为”，文档 A 提到了 10 次，文档 B 提到了 1 次，显然 A 更有可能是你想要的。
- **BM25 的优化**：它引入了“词频饱和度”。在 TF-IDF 中，出现 100 次的分数远高于 10 次；但在 BM25 中，当次数达到一定程度后，分数的增长会迅速放缓。因为提到 10 次和提到 100 次，相关性其实差不了太多。

### 2. 逆文档频率 (IDF - Inverse Document Frequency) 💎

**逻辑**：越稀有的词，权重越高；越普遍的词，权重越低。

- **原理**：假设你搜索“华为手机”。“手机”这个词在几万个文档里都有，它不值钱；而“华为”这个词相对较少，它更能代表你的搜索意图。
- **计算**：ES 会统计包含某个词的文档总数。文档数越多，该词的 IDF 分数就越低。

### 3. 字段长度归一化 (Field Length Norm) 📏

**逻辑**：在短字段中匹配到关键词，比在长字段中匹配到更有价值。

- **原理**：如果你搜“手机”。
  - 文档 A 的标题是：`“手机”`（共 2 个字）。
  - 文档 B 的标题是：`“这是一款性能强劲、外观精美的智能手机”`（共 20 多个字）。
- **判定**：文档 A 的匹配密度更高，所以 A 的分数会更高。

### 🎓 老师的代码分析：Search-Service 中的应用

在我们的项目中，搜索功能通常在 `search-service` 模块。你可以观察一下搜索请求的代码：

```java
// 伪代码示例
SearchResponse response = client.search(s -> s
    .index("items")
    .query(q -> q
        .match(m -> m
            .field("name")
            .query("华为手机")
        )
    )
);
```

当你执行这段代码时，ES 后台会自动触发 BM25 计算。如果你想手动干预排名（比如：给“华为”品牌加权，或者让“广告位”商品排在前面），你会用到 **Function Score Query**。

#### 什么是 Function Score？🛠️

这是大厂最常用的手段。它允许你在 BM25 的基础分上，乘上一个特定的权重。比如：

- **业务加权**：如果该商品是“自营”，分数乘以 1.2。
- **销量加权**：销量越高，分数越高。
