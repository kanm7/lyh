Feign 的工作原理: 为什么在 hm-api 里定义接口，cart-service 就能调用 item-service？(动态代理、Ribbon/LoadBalancer 负载均衡)。

### 注册中心与配置中心：Nacos

项目中的作用:

•注册中心: cart-service、item-service 等启动时，会把自己的 IP 和端口注册到 Nacos。当 cart-service 需要调用 item-service 时，会去 Nacos 找“item-service 在哪里”。

•配置中心: 将 application.yaml 中的配置（如数据库密码、开关配置）放到 Nacos 上统一管理，实现配置的热更新，不用重启服务。

Q: Nacos 如何实现配置热更新？

A: 客户端通过长轮询（Long Polling）监听服务端配置变化。Spring Boot 中配合 @RefreshScope 注解实现 Bean 的动态刷新。

### 服务网关：Spring Cloud Gateway

项目中的作用:

•统一入口: 前端请求不直接发给 cart-service，而是发给网关，网关再转发。

•路由转发: 根据 URL 路径（如 /cart/**）找到对应的微服务。

•鉴权: 在网关的 GlobalFilter 中校验用户 Token，解析用户信息并放入请求头传递给后端服务。

Q: 网关怎么做登录校验？

•A: 定义一个全局过滤器（GlobalFilter），拦截所有请求。解析请求头中的 Token（如 JWT），验证合法性。如果合法，放行；如果不合法，返回 401。

### 远程调用：OpenFeign

项目中的作用:

让微服务之间的 HTTP 调用变得像调用本地方法一样简单。你只需要定义一个接口（如 ItemClient），加个注解，Feign 就会自动生成代理类发起请求。

Q: Feign 的底层原理是什么？

•A:

1.动态代理: 项目启动时，Feign 会扫描 @FeignClient 注解，生成动态代理类。

2.构造请求: 当调用接口方法时，代理类会根据注解（@GetMapping 等）拼接 URL 和参数。

3.发送请求: 底层默认使用 JDK 的 HttpURLConnection，但在生产环境中通常会配置连接池（如 Apache HttpClient 或 OKHttp）来优化性能。

•Q: 你们项目中 Feign 做了哪些优化？

•A:

1.配置了连接池（替代默认的 URLConnection）。

2.配置了日志级别（开发环境 FULL，生产环境 BASIC）。

### 负载均衡：Spring Cloud LoadBalancer

项目中的作用:

•当 item-service 有 3 个实例（启动了 3 个服务）时，Feign 需要决定调用哪一个。LoadBalancer 负责这个选择过程。

Q: 常见的负载均衡策略有哪些？•A: 轮询（Round Robin，默认）、随机（Random）、权重（Nacos 权重）、最少连接数等。

### 服务保护：Sentinel

项目中的作用:

•限流: 限制 hm-gateway 或热点接口的 QPS，防止流量激增把服务打挂。•

熔断降级: 当 item-service 挂了或者响应太慢，cart-service 调用它时会自动熔断（快速失败），而不是一直等待，避免雪崩效应。

面试官常问:

•Q: 什么是服务雪崩？怎么解决？

•A: 服务 A 调用 服务 B，B 挂了，导致 A 的线程阻塞耗尽，A 也挂了，最终整个系统瘫痪。解决办法是使用 Sentinel 做超时处理、线程隔离和熔断降级。

•Q: Sentinel 和 Hystrix 的区别？

•A: Hystrix 也就是断路器，主要做隔离和熔断。Sentinel 功能更强，支持流量整形（预热、排队）、热点参数限流，且有可视化的控制台，对代码侵入性更小。

### 分布式事务：Seata

项目中的作用:

•下单业务涉及：创建订单（Trade库） + 扣减库存（Item库）。这两个操作必须同时成功或同时失败。Seata 负责协调这个全局事务。

面试官常问:

•Q: 你们用的 Seata 的哪种模式？

•A: 一般常用 AT 模式（默认模式）。它对业务代码无侵入，Seata 自动生成 undo_log（回滚日志）来实现回滚。•Q: AT 模式的原理？

•A: 一阶段：执行业务 SQL 并提交本地事务，同时记录 undo_log。二阶段：如果全局提交，异步删除 undo_log；如果全局回滚，根据 undo_log 生成反向 SQL 执行回滚。

### 消息队列：RabbitMQ

项目中的作用:

•异步解耦: 如下单成功后，发送消息给 MQ，通知购物车服务清空购物车，通知短信服务发短信。这样下单接口响应更快。

•面试官常问:

•Q: 消息可靠性怎么保证（怎么防止丢消息）？

•A:1.发送端: 开启 Publisher Confirm 机制。2.MQ端: 开启持久化（Exchange, Queue, Message 都持久化）。3.消费端: 关闭自动 ACK，业务处理成功后手动 ACK。

•Q: 消息积压怎么办？

•A: 增加消费者数量；或者临时开启一个新的 Topic，把积压消息转发到新 Topic，用更多机器快速处理。

### 如果面试官问我，远程服务失败了，那么该如何处理，我该如何回答

1.重试

2.如果重试仍然失败或者服务挂掉了，会触发sentinel的熔断，之后走Feign的降级处理逻辑

3.设置业务层面的兜底方案，记录失败日志或者发送一条消息到死信队列，后续通过定时任务或者人工干预处理，当然也可以通过seata的分布式事务进行回滚或者通过rabbitmq进行异步补偿，保证数据的最终一致性。

### 面试官如果问我jwt令牌的原理

它的核心原理是**‘无状态认证’**。

 一个 JWT 字符串由三部分组成：

Header：声明类型是 JWT 以及使用的签名算法（如 HS256 或 RS256）。

Payload：存放有效信息

Signature：服务器用‘Header + Payload’配合一个密钥 (Secret)，通过算法生成签名。

#### 认证流程是：

1.用户登录成功后，服务器生成一个 JWT 返回给客户端。

2.客户端将 JWT 存在 LocalStorage 或 Cookie 中。

3.后续请求时，客户端在 HTTP Header (Authorization: Bearer <token>) 中带上这个 Token。

4.网关或服务端收到后，用密钥校验签名。如果签名对得上，说明数据没被篡改，且未过期，就放行，不需要去查数据库。”

### jwt怎么解决session的一些问题

Session: Session 数据存在服务器内存中。如果是集群部署（比如 cart-service 部署了 3 台），用户在 A 机器登录，下次请求转发到 B 机器，B 机器没有 Session，用户就得重新登录。虽然可以用 Session 共享（如存 Redis），但增加了系统复杂度。

JWT: JWT 是无状态的。服务器不存 Token，Token 自身包含了所有用户信息。只要服务器有密钥，任何一个微服务节点都能校验 Token，天然支持分布式集群。

### jwt的加密算法了解吗，我该如何回答

对称加密：签发token和校验token用的是同一个密钥

非对称加密：一对私钥和公钥。用私钥签发token，用公钥校验token

### “那 JWT 有什么缺点吗？” 或者 “如果我想让用户强制下线怎么办？”

“JWT 最大的优势是无状态，但这也是它最大的缺点。一旦签发，在过期之前无法废止。 如果需要实现‘踢人下线’或‘修改密码后强制登出’，我们可以利用redis，把需要作废的 Token ID （在签发的时候根据雪花算法来生成，全局唯一，存入payload中）存入 Redis，设置过期时间。每次校验时多查一次 Redis（但这稍微牺牲了一点无状态的优势）。

### 那jwt和redis怎么结合来使用

通常 JWT 是无状态的，不需要存 Redis，但为了解决 JWT 的一些固有缺陷（比如无法主动注销），我们在实际项目中（包括 hmall 这种）经常会把 JWT 和 Redis 结合使用。

#### JWT 签发后，在过期前一直有效。如果用户注销（退出登录），或者用户账号被封禁，如何让原本合法的 Token 立即失效？

“虽然 JWT 本身是无状态的，但在我们项目中，为了实现用户注销和强制下线功能，我们结合了 Redis。

具体做法是：

​	1.注销时：当用户点击‘退出登录’，后端会把这个 Token 的 ID (JTI) 或者整个 Token 字符串存入 Redis，并设置过期时间（剩余有效期）。

​	2.校验时：网关在校验 Token 签名有效后，会多做一步操作——去 Redis 查一下这个 Token 是否在‘黑名单’里。如果在，就拒绝访问。

这样既保留了 JWT 的大部分优势，又解决了无法主动失效的问题。”

#### 缓存用户信息 (性能优化)

JWT 的 Payload 里虽然能存信息，但不能存太多（否则 Header 太大，传输慢）。如果业务需要频繁用到用户的详细信息（如权限列表、会员等级），每次都去查数据库太慢。我们还用 Redis 来缓存用户信息，这样避免了每次请求都去查数据库，大大提升了并发性能。”

### “为什么用 ES 搜东西比 MySQL 快？/ ES 的原理是什么？”核心原理：倒排索引 (Inverted Index)

“MySQL 采用的是正向索引（B+树），它是根据 ID 找数据。如果要根据‘标题’模糊搜索（LIKE %手机%），索引会失效，导致全表扫描，性能非常差。

而 ES 采用的是倒排索引。 它的原理是：在存储数据时，先对内容进行分词（比如把‘华为手机’分成‘华为’、‘手机’）。然后建立一个映射表，记录每个词条 (Term) 出现在了哪些文档 (Document) 里。

搜索时：比如搜‘手机’，ES 直接去查倒排索引，瞬间就能找到包含‘手机’的所有文档 ID，而不需要遍历所有数据。这就好比查字典，我们是根据‘偏旁部首’（词条）直接找到‘字’（文档），而不是一页页翻书。”

### 面试官问：“你们项目中哪些地方用到了 ES？”

1.全文检索：用户在搜索框输入关键词（如‘手机’），我们需要从海量商品中快速搜出匹配的商品。

2.高亮显示：搜索结果中，把用户搜的关键词（如‘手机’）标红显示，提升体验。

3.聚合统计 (Aggregation)：这是 ES 的强项。比如用户搜‘手机’，我们需要在侧边栏展示‘品牌列表’（华为、小米）、‘价格区间’等。这需要对搜索结果进行聚合分析，MySQL 做这个非常慢，而 ES 非常快。”

### 难点：MySQL 和 ES 数据怎么同步？(必考题)

我们项目中采用了异步通知的方案，利用 RabbitMQ 来实现最终一致性。
具体流程是：

​	1.写操作：当运营人员在后台（item-service）修改了商品信息或上下架商品后，会写 MySQL 数据库。

​	2.发消息：数据库更新成功后，发送一条消息（包含商品 ID）到 RabbitMQ 的 Exchange。

​	3.消费消息：搜索服务（search-service）监听这个队列。收到消息后，根据 ID 去查最新的商品数据，然后调用 ES 的 API 更新索引库。

为什么不用同步调用？ 因为同步调用（OpenFeign）会增加业务耗时，且如果 ES 挂了，会影响正常的商品修改业务。用 MQ 解耦更稳定。”

### 性能坑：深分页问题 (Deep Paging) / “如果用户搜出来 10 万条数据，想看第 10000 页，ES 性能会怎样？”

“这会产生深分页问题。 ES 的分页是 from + size 模式。如果要查第 10000 页（每页 10 条），ES 需要在每个分片上先查出前 100010 条数据，汇总到协调节点，排序后截取最后 10 条。这会消耗大量内存和 CPU。

解决方案：

1.业务限制：像百度、京东一样，限制用户只能看前 100 页。普通用户根本不会翻到第 10000 页。

2.Search After：如果是后台导出数据需要全量遍历，我们会用 search_after 方案，它是根据上一次查询的最后一条数据的值来往后查，性能更好，但不适合随机跳转分页。” （思想可以类比zset的score查询，但是不完全一致）
