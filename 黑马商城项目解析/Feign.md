## Feign

### 1. 核心流程：面试必背四步法 🏆

如果面试官问“Feign 是怎么工作的？”，你可以这样有条理地回答：

1. **“Feign 的核心是通过 JDK 动态代理来模拟接口实现，实现‘声明式’的服务调用。”**

   1. **启动阶段**：Spring 会扫描标注了 `@FeignClient` 的接口，并利用 **JDK 动态代理** 为其生成一个代理对象，接管接口调用。
   2. **解析阶段**：当程序调用接口方法时，代理对象会拦截请求，通过 **Contract（契约）** 组件解析方法上的 Spring MVC 注解，确定请求的路径和参数。
   3. **构建阶段**：Feign 将参数通过 **Encoder（编码器）** 序列化为 JSON，并结合 **负载均衡器（LoadBalancer）** 从注册中心选出一个具体的服务实例 IP，构造出完整的 HTTP 请求。
   4. **执行阶段**：最后由底层的 **Http Client**（如 OkHttp 或 HttpClient）正式发送请求，并在获取响应后通过 **Decoder（解码器）** 将结果反序列化为 Java 对象返回给调用者。

   ### 关键名词快速记忆表 🧠

   | **组件名称**          | **核心职责**                                                 |
   | --------------------- | ------------------------------------------------------------ |
   | **JDK 动态代理**      | **“变魔术”**：给没有实现类的接口凭空造出一个能干活的对象。   |
   | **连接池**            | **就是**“**提前准备好一堆连接**，**大家重复用**”，**避免每次都重新创建连接**。 |
   | **Contract (契约)**   | **“翻译官”**：把 `@GetMapping` 等注解翻译成 URL 路径。       |
   | **Encoder / Decoder** | **“打包员”**：负责 Java 对象与 JSON 数据之间的转换。         |
   | **Load Balancer**     | **“导航仪”**：在多个服务器中选出一个不挤的 IP 路径。         |
   | **HttpClient**        | **是一个 HTTP 请求客户端。**Java程序  →  发送 HTTP 请求  →  服务器 **HTTP底层是TCP连接，创建TCP连接成本高，所以需要连接池复用连接。** |

### 2.“如果线上的 Feign 调用突然报错了（比如 404 或超时），你该如何利用这些组件的知识去排查？”

### 🛠️ Feign 故障排查“三板斧”

当一个 Feign 接口没调通时，我们可以按以下顺序“点名”排查：

#### 1. 检查“翻译官” (Contract)：是不是路带错了？（404 错误）

- **现象**：报 404 Not Found。
- **排查点**：
  - 看看你在接口上写的 `@GetMapping("/users/{id}")` 路径对不对。
  - 检查参数名是否匹配。
  - **面试话术**：“首先我会排查 **Contract（契约组件）**，看它解析出来的请求路径和方式是否与服务提供方的接口定义一致。”

#### 2. 检查“打包员” (Encoder/Decoder)：是不是货装错了？（400/500 错误）

- **现象**：报 400 Bad Request 或服务端报 JSON 解析异常。
- **排查点**：
  - 是不是 Java 对象里的字段名和对方要求的不一样？
  - 是不是少传了某个必填字段？
  - **面试话术**：“我会关注 **Encoder（编码器）**。有时候因为字段序列化的问题，导致发出去的 JSON 格式不符合对方要求。我会开启 Feign 的**详细日志（Logger.Level.FULL）**，观察最终发出的报文实体。”

#### 3. 检查“跑腿小哥” (HttpClient/LoadBalancer)：是不是人跑丢了或跑慢了？（超时错误）

- **现象**：报 Read Timeout 或 Connect Timeout。
- **排查点**：
  - **LoadBalancer（负载均衡）**：是不是选到了一个已经挂掉的服务器实例？
  - **HttpClient 配置**：默认的超时时间通常很短（比如 1 秒）。如果对方接口比较慢，就会直接断开。
  - **面试话术**：“我会排查 **HttpClient** 的超时配置。如果业务逻辑较重，我会适当调大 `readTimeout`；同时检查负载均衡器是否获取到了最新的服务列表，排除死节点的影响。”

### 🎓 老师的面试心法：善用“日志”这个放大镜

在面试中，如果你能主动提到**“开启 Feign 日志”**，面试官会觉得你很有实战经验。

- **知识点**：Feign 默认不打印任何日志。
- **实战做法**：在配置文件中把对应的 Feign 接口包路径设为 `DEBUG` 级别，并在代码里配置 `Logger.Level.FULL`。
- **效果**：这样你就能在控制台清清楚楚地看到：**请求头是什么、JSON 长什么样、对方回了什么。** 绝大多数问题在日志面前都无所遁形。

### 3.Feign结合Sentinel

在微服务中，最怕的不是某个服务挂了，而是它“半死不活”（响应极慢）。这会导致调用它的服务一直等待，最后把整个系统的线程池耗尽，引发**级联失效（Cascading Failure）**，也就是俗称的“雪崩”。	

| **术语**                    | **形象类比**   | **技术定义**                                                 |
| --------------------------- | -------------- | ------------------------------------------------------------ |
| **熔断 (Circuit Breaking)** | **保险丝** 🔌   | 当目标服务错误率达到阈值（比如 50%），Sentinel 直接切断连接，不再发请求，保护调用方。 |
| **降级 (Fallback)**         | **备选方案** 🍱 | 当请求失败、超时或被熔断时，不去报错，而是执行一段预设的“备选代码”，返回一个友好值。 |

### Feign 集成 Sentinel 的工作原理 ⚙️

在代码层面，Feign 是如何开启这项能力的？

- **名词解释：Fallback (回退)** 🔄 当远程调用失败时，程序会自动跳转到一个预先写好的“备选类”中执行。

**老师的“代码式”分析：**

1. **开启开关**：在 `application.yml` 中设置 `feign.sentinel.enabled = true`。
2. **编写备选类**：创建一个实现类，重写接口方法。

```java
// 1. 定义接口，并指定失败后的“备选类” (Fallback)
@FeignClient(value = "user-service", fallback = UserClientFallback.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    String getUserName(@PathVariable("id") Long id);
}

// 2. 编写备选逻辑
@Component
class UserClientFallback implements UserClient {
    @Override
    public String getUserName(Long id) {
        // 当远程服务挂了，这里返回一个默认值，保证业务不中断
        return "用户服务暂时不可用（系统降级）";
    }
}
```

### 3. 面试高频题：Sentinel 的隔离策略 🚦

面试官会问：“Sentinel 是怎么限制流量，不让某个接口把整个服务拖死的？”

这里有两个专业名词，你一定要记准：

- **线程池隔离 (Thread Pool Isolation)**： 给每个接口分配一个专门的“小房间”（线程池）。如果 A 接口爆了，只会占满它自己的房间，不会影响 B 接口。缺点是**开销大**。
- **信号量隔离 (Semaphore Isolation)**： Sentinel **默认采用**这个。它就像给接口发“入场券”（计数器）。只有拿到券的请求才能执行。如果 10 张券发完了，后面的请求直接拒绝。优点是**轻量、性能极高**。

### 4. 故障排查实战：如何确定熔断阈值？ 🕵️

面试官问：“你项目里的熔断阈值（比如 5s 内错误率 50%）是怎么定出来的？”

**高分回答：**

> “这个值不是拍脑袋定的。我们首先通过 **JMeter 进行压测**，观察服务在正常负载下的平均响应时间（RT）。如果正常 RT 是 200ms，那么我们可以把超时阈值设为 1s，熔断阈值设为错误率 50%。上线后，我们会通过 **Sentinel 控制台** 实时监控，根据真实的流量波动进行动态微调。”
