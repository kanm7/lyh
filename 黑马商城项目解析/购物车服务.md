## 购物车服务

在电商系统中，购物车是连接“搜索”与“下单”的关键桥梁。虽然功能看起来只是简单的增删改查，但其背后的架构选择非常有讲究。

------

### 1. 存储方案的抉择：为什么是数据库 (DB)？ 💾

在很多技术博客里，大家习惯用 Redis 存储购物车，但在中大型电商项目中，使用 **MySQL 数据库**（配合索引优化）其实是更稳健的选择。

- **多端同步** 📱💻：用户在手机上加入购物车的商品，回到电脑上登录也得能看到。数据库作为持久化存储，天生适合这种跨设备的数据同步。
- **数据价值** 📊：购物车里的商品代表了用户的“购买意图”。如果存放在 Redis 中，万一内存溢出或过期，这些珍贵的意图数据就丢了。而存入数据库，我们可以方便地进行后续的营销分析（比如给购物车有商品但没买的用户发优惠券）。
- **性能平衡** ⚖️：虽然 Redis 更快，但购物车操作通常是针对 `user_id` 的。给 `user_id` 加上索引后，MySQL 的查询性能足以支撑千万级用户的日常操作。

### 2. 购物车的“生命周期”与异步清理 🧹

当用户点击“提交订单”后，购物车里的商品就该“退场”了。这里我们采用了**异步解耦**的方案：

1. **交易服务 (Trade Service)**：创建订单成功后，顺手往 RabbitMQ 发个“消息”。 ✉️
2. **购物车服务 (Cart Service)**：它的 `deleteItemsListener` 就像个监听雷达。 📡
3. **处理逻辑**：一旦收到消息，它会立刻从数据库中删除对应的商品 ID。

这样设计的好处是：下单不需要等待购物车清理干净就能直接告诉用户“下单成功”，系统的响应速度飞快。 🚀

------

### 🎓 老师的引导提问

我们来看一个具体的代码逻辑：在 `CartServiceImpl` 中，当用户**重复添加**同一个商品到购物车时，我们通常不会新增一行数据，而是执行 `UPDATE` 操作增加 `num`（数量）。

**请思考一下：** 如果在极高并发的情况下，用户连续快速点击了两次“加入购物车”，而我们的代码逻辑是“先查询是否存在，再决定新增还是修改”，这会产生什么样的数据安全问题？🤔（提示：你可以往“数据库并发冲突”的方向想想。）

想象一下，你正在用手机飞快地点击“加入购物车”，由于你手速极快，两个请求**同时**到达了服务器。

### 1. 案发现场：什么是“并发冲突”？💥

假设你的购物车里原本**没有**这件商品。

- **请求 A（线程 1）**：先去数据库查，“用户张三有这个商品吗？”数据库回：“没有”。
- **请求 B（线程 2）**：紧跟着也去查，“用户张三有这个商品吗？”由于请求 A 还没来得及存进去，数据库依然回：“没有”。
- **请求 A**：既然没有，那我就执行 `INSERT`（新增一条记录，数量为 1）。
- **请求 B**：我也收到了“没有”的消息，那我也执行 `INSERT`（再新增一条记录，数量为 1）。

**结果**：你的购物车里竟然出现了**两条一模一样**的商品记录！这在业务逻辑上是不允许的。

这种情况在技术上有一个专业的名称，叫 **“竞态条件”（Race Condition）**。

------

### 2. 怎么解决这个“双胞胎”问题？🛡️

在我们的 `cart-service` 项目中，为了防止这种乱子，通常有两种“防线”：

#### 第一道防线：数据库唯一索引（硬约束）

我们在数据库表里，给 `user_id`（用户 ID）和 `item_id`（商品 ID）加一个**联合唯一索引**。

- 这样，当请求 B 尝试插入第二条一模一样的记录时，数据库会直接报错：“嘿！已经有一条了，不准再插！”

#### 第二道防线：原子化更新（代码技巧）

我们不采用“先查再改”的笨办法，而是直接用一条 SQL 搞定： `INSERT INTO cart ... ON DUPLICATE KEY UPDATE num = num + 1`

- **大白话**：这条 SQL 的意思是，“去插这条数据，如果发现已经存在了（触发了唯一索引），就直接把数量加 1”。这样无论多少个请求并发，最后的结果都是准确的。
