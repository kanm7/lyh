## 网关知识点

### 核心要点总结 📝

1. **统一鉴权**：由 `hm-gateway`（网关）统一校验 JWT 令牌，避免每个微服务重复写校验逻辑。
2. **信息传递**：网关校验通过后，将用户信息放入 HTTP Header 传递给下游微服务。
3. **线程隔离存储**：微服务内部通过 `Interceptor`（拦截器）提取 Header 信息，并存入 `ThreadLocal`，确保业务代码随取随用且线程安全。

### 第一站：网关的“安检处” —— `AuthGlobalFilter` 🛡️

在 `hm-gateway` 模块中，`AuthGlobalFilter` 实现了 `GlobalFilter` 接口。它的作用就像学校大门的保安。

- **分析逻辑**：
  - **判定白名单**：首先检查请求路径是否在排除名单内（如登录接口）。如果在，直接放行。
  - **JWT 校验**：如果不在白名单，则从请求头或 Cookie 中提取 JWT 令牌。使用 `JwtTool` 进行解析和验证。
  - **信息转发**：校验通过后，网关会将解析出的 `userId` 存入请求头（例如名为 `user-info`），转发给下游服务。

> **名词解释**：**JWT (JSON Web Token)** 🎫 是一种用于双方之间安全传输信息的简洁、URL 安全的令牌。它由头部、负载和签名组成，网关只需通过密钥就能验证令牌是否被篡改。

### 第二站：微服务的“内部接收员” —— `UserInfoInterceptor` 📥

当请求离开网关到达具体的服务（如 `cart-service`）时，服务需要一种高效的方式来获取当前登录的用户是谁。这时 `hm-common` 模块中的 `UserInfoInterceptor` 就上场了。

- **分析代码**：
  - 它实现了 `HandlerInterceptor`。在业务逻辑执行前（`preHandle`），它会尝试从 Header 中读取网关传来的 `user-info`。
  - 读取到后，它会将用户信息存入 `UserContext`。

### 第三站：数据的“私人储物柜” —— `UserContext` & `ThreadLocal` 🧠

这是面试官最喜欢深挖细节的地方。

Java

```java
public class UserContext {
    private static final ThreadLocal<Long> THREAD_LOCAL = new ThreadLocal<>();
    // ... 存取方法
}
```

- **为什么要用 ThreadLocal？**
  - **线程隔离**：每个请求在 Tomcat 中都是由独立的线程处理的。`ThreadLocal` 为每个线程提供了一个独立的变量副本，互不干扰。
  - **减少侵入**：你不需要在 Service 层的每一个方法参数里都传一个 `userId`。只要在同一个线程内，任何地方调用 `UserContext.getUser()` 都能拿到用户 ID。

面试官可能会问：“**在微服务中使用 ThreadLocal 有什么潜在风险？**”

1. **内存泄漏**：`ThreadLocalMap` 的 Entry 是弱引用，但 Value 是强引用。如果线程不销毁（如线程池），且没有手动执行 `remove()`，Value 就可能无法被 GC 回收。
2. **异步传递失效**：如果你在业务逻辑里开了子线程（如 `CompletableFuture`），子线程是拿不到父线程 `ThreadLocal` 里的信息的。
